	Регимся на сайте github.com, создаем репозиторий, копируем путь к репозиторию
	Устанавливаем с сайта http://www.syntevo.com программу smartgit, делаем команду клон
и вставляем путь к удаленному репозиторию. Эта команда выполняется единоразова
(перемещает все что есть на репозитории github.com на локальный репозиторий
	Комманда Commit сохраняет изменения на моем локальном репозитории
	Чтобы разместить в репозиторий github.com нужно выполнить команду Push
	тобы получить свежие изменения с удаленного репозитория github.com на локальный репозиторий нужно
выполнить команду Pull
	Если изменения еще не закоммитены можно сделать Discard и отменить последние изменения в 
файле.
	Если изменения закоммитены то заходим в Log, выбираем что именно хотим откатить 
и нажимаем команду Revert&Commit (помогает откатить уже существующие коммиты)
	В логе можно удалить ветку то что было сделано после какогото коммита 
выбрав команду Reset
	Правило гита: у Вас под рукой всегда должа быть рабочая версия программы,
стабильная протестированная, которую Вы в любой момент готовы отдать своим заказчикам.
	Если вы разрабатываете дополнительно какуюто новую фичу, вы должны разрабатывать 
ее параллельно, не трогая стабильную версию, до тех пор пока не будете в ней уверены.
	Как только мы начинаем работу над новой фичей мы создаем новую ветку.
	После создания ветки кликаем на нее два раза делаем Check Out Branch 
(теперь мы работаем в новой ветке!)
	Если вы закончили работать над новой фитчей нужно подтянуть все изменения из ветки мастер
в ветку с новой фитчей(если были изменения в ветке мастер)это делается с помощью команды Merge
кликая на ветке мастер. Все изменения в мастере подтянулись в ветку с альтернативной веткой с новой фичей,
затем проверяем программу в ветке с новой фитчей чтобы программа не поломалась. Переходим в ветку мастер 
с помощью комманды check out и подтягиваю все изменения из ветки с новой фичей в ветку мастер.  
	Ошибки разрешаем в Conflict Solver вручную
	Ошибка №1 боязнь создавания новых веток. Когда вы добавляете новый функционал в 
программуили добавляете новую фичю то нужно создавать новую ветку, 
отрабатывать код в этой ветке и только потом сливать в ветку мастер.
Ошибка №2 добавлять в репозиторий бинарники(exe и т д)
чтобы они не добавлялись выполнить команду Ignor
Ошибка #3 считать что в ветке в которой вы разрабатываете д б исключительно код 
относящийся к новой фичи.
Если что то было удалено в новой ветке,при разработке новой фичи, то при слиянии
с веткой мастер эти строки (файлы) тоже удалятся
Ошибка №4 делать очень большие коммиты, пример разработчик работает целый день или несколько 
дней и делает только один коммит, это не правильно, нужно делать при кажом 
связанном логическом изменении текста программы делать коммит.
Находясь в логе можно не сливая полностью ветки слить только изменения сделанные в одном коммите
выбрав нужный коммит и выполнить комманду Cherry-Pick тем самым вы выбирите коммит который 
вы выбрали в логе

делаем команду git clone "путь к удаленному репозиторию"
git status команда показывает есть ли какието изменения в нашем репозитории
git add
git commit -a -m "Внес изменения такие то..."
git push
git log
git branch смотрим в какой ветке мы находимся
git branch love создаем новую ветку love
git branch -a 
git add love.txt добавляем в систему контроля версий файл love.txt
git status видим что добавился новый файл
git commit -a -m "добавили новый файл"
git push origin love отправляем ветку love в удаленный репозиторий
git checkout master переключаемся в ветку мастер
git branch -a проверяем что мы находимся в ветке мастер
git merge love сливаем ветки мастер и love
git push original отправляем в удаленный репозитори
git branch -d love удаляем ветку love 
git push original --delete love удаляем ветку love в удаленном репозитории
